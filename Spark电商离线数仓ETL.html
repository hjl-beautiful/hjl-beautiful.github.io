业务背景

电商原始交易数据通常是脏、散、未建模的流水，直接分析既慢又不规范。离线数仓的价值在于：用分层架构把"原始数据"逐步加工为"可信任、可复用、可分析"的指标资产，支撑下游 BI、用户分群与经营分析。

核心问题

能否用 Spark 大数据引擎，把 39 万条英国电商交易数据清洗、建模为规范的四层数仓，并产出 RFM 分群与复购率等可复用指标？

→

ADS

关键设计

维度建模：以用户、商品、时间为维度构建星型/雪花模型，保证指标口径一致；列式存储：中间结果落 Parquet，提升查询效率；数据质量：在 DWD 层做空值、异常值、重复订单校验。

核心产出

  • RFM 用户分群：在 DWS 用户主题宽表上计算 Recency/Frequency/Monetary，输出分群标签；
  • 复购率分析：计算 65.51% 用户复购率，定位高复购品类与召回人群；
  • 可复用指标层：ADS 层沉淀口径统一的指标，供下游 BI 直接取数，避免重复计算。

调度与可靠性

用 Airflow 编排各层 ETL 任务的依赖与重试，配合数据质量校验，保证每日跑数的稳定性与可观测性——这是数据开发岗的核心工程能力体现。

RFM SQL 复用（跨项目验证）

本项目的 RFM 分群逻辑已沉淀为可复用的 rfm_analysis.sql，与电商用户分析项目共用同一套 SQL，并在真实 UCI 数据（406,829 行）上验证通过：

  • 口径一致：NTILE 五分位 + RANK 选品 + LAG 复购间隔，与本数仓 ADS 层 RFM 指标同构；
  • 互相印证：SQL 复购率 65.58% 与本项目口径 65.51% 高度吻合，验证分群逻辑正确；
  • 数据闭环：体现"数仓产出 → SQL 工程化 → 跨项目复用"的标准做法，正是企业数据开发提倡的能力。

项目亮点

  • 工程规范：严格四层分层 + 维度建模，体现数据开发的标准方法论；
  • 大数据栈：PySpark / Spark SQL / Parquet 贴合企业数据开发技术栈；
  • 可运维：Airflow 调度 + 质量校验，从"能跑"到"可靠地跑"；
  • 业务衔接：数仓产出直接喂给 RFM / 复购分析，对接电商用户分析项目，形成数据闭环。

局限性 & 迭代方向

  • 实时性：当前为离线批量，后续可补 Spark Streaming / Flink 近实时链路；
  • 规模验证：39 万条属中小样本，需补充更大体量下的分区、倾斜处理与性能调优；
  • 指标体系：ADS 层指标可随业务扩展（留存、漏斗、GMV 拆解），并接入指标管理平台统一口径。
