业务背景

九寨沟作为世界自然遗产景区，客流高度集中于节假日与旅游旺季，存在两类运营痛点：

  • 超载风险：单日客流超过核定最大承载量时，既威胁游客安全与体验，也违反景区限流规定；
  • 运力错配：摆渡车、餐饮与住宿按固定排班配置，无法匹配客流峰谷，导致高峰滞留、平峰空驶。

核心问题

能否用历史公开日度客流数据，提前 7 日预测未来客流并给出置信区间，支撑「预约限流 + 运力调度 + 商业备货」决策？

数据与方法

  • 数据：九寨沟景区公开日度客流数据，1,869 天真实记录，日度粒度。
  • 特征工程：40 维，覆盖时间特征（年/月/周/季节）、节假日标记、滞后项（t-1/t-7/t-30）、滚动统计（7/30 日均值、标准差）。
  • 建模：XGBoost 回归，对比随机森林、线性回归基线；按时间切分训练/测试，避免未来信息泄漏。
  • 可解释性：SHAP 输出特征贡献，识别「节假日、去年同期、近期滚动均值」为 Top 驱动因子。
  • 部署：Streamlit 四页面应用（预测 / 置信区间可视化 / 因素分析 / 明细），已上线可演示。

模型效果

XGBoost 在测试集上取得 R² = 0.9665、MAPE = 5.1%，优于随机森林与线性回归基线，输出未来 7 日客流点预测与 90% 置信区间。日度客流具有强季节 + 节假日驱动 + 强自相关特性，配合滞后/滚动特征后高 R² 合理。

业务闭环（项目亮点）

预测不是终点，落地才是价值。系统把预测值翻译成运营可用的指标与动作：

① 架构流程

预测
XGBoost 输出未来 7 日客流 + 90% 置信区间
→
预警
计算日饱和度 S，S>0.9 触发超载预警清单
→
调度
预约限流 · 运力排班 · 商业备货 · 错峰引导

② 业务指标

  • 日饱和度 S = 预测客流 / 核定日最大承载量 T（T 按景区最新规定参数化）；
  • 超载预警：S > 0.9 的日期清单，提前 7 日锁定；
  • 预测区间覆盖率 PICP：真实值落在 90% 置信带内的比例（回测应 ≈ 90%，用于校准可信度）。

③ 四类运营策略

  • 预约限流：对预警日提前开放分时段预约、设置日售票上限，平滑单日负荷；
  • 运力调度：摆渡车 / 观光车按预测峰谷比动态排班，降低滞留与空驶；
  • 商业备货：餐饮、住宿、便利店基于 7 日预测做备货与人员排班；
  • 错峰引导：发布错峰游览建议，将高饱和日游客向相邻低饱和日迁移。

可量化收益（按模型误差推算）

模型 MAPE=5.1%，可提前 7 日锁定约 90% 以上的高峰日，显著降低超载误判；运力按峰谷比动态排班，理论上可减少平峰空驶与高峰滞留（可在试点后实测量化）。

说明：以上为方法论层面的收益估算，非已落地实测数字，面试中如实表述更稳妥。

Power BI 运营看板（数据可视化延伸）

在预测模型之外，额外构建了一套 Power BI 星型模型，把日度客流事实表与日期维、法定节假日维关联，支撑运营人员自助分析与汇报：

  • 星型模型：fact_visitors（日度客流事实）+ dim_date（日期维）+ dim_holiday（145 天法定节假日维），dim_date / dim_holiday 均 1 对多关联 fact_visitors[date]；
  • 下钻能力：支持按法定节假日 / 年份 / 月 / 星期维度下钻客流分布，定位节假日驱动的峰值规律；
  • DAX 度量：用 DAX 计算同比、节假日客流占比、峰值日饱和度等运营指标，可解释、可复用；
  • 交付物：仓库内提供 fact_visitors.csv / dim_date.csv / dim_holiday.csv 三份 CSV + Power BI 执行清单与看板规划文档，可直接在 Power BI Desktop 打开建模型。

为什么补 Power BI

预测给出"未来多少"，BI 回答"过去为什么、现在怎么分布"。两者互补：XGBoost 做预测与预警，Power BI 做日常运营下钻与复盘，覆盖"预测 → 预警 → 调度 → 复盘"完整链路，更贴合数据分析 / BI 岗要求。

局限性 & 迭代方向

  • 高 R² 的来源：已用时间切分、无未来信息泄漏；强季节 + 节假日 + 自相关是高 R² 的合理解释。
  • 置信区间校准：90% 区间需回测 PICP ≈ 90%，若偏窄将引入分位数回归 / Conformal Prediction 改进。
  • 外部冲击：疫情、政策限流、极端天气等黑天鹅未纳入，模型在突变期失效。
  • 收益量化：当前为估算，计划接入真实调度记录做 A/B 对照，把"估算"变为"实测"。
